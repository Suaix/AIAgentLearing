## 各维度详评

### A - 概念理解深度：L3 🟢 (3.0)

**证据**：
- Step 1 Q1：准确识别信号稀释效应——"整篇编码的相似度几乎为零"的核心原因是信息被其他主题稀释
- Step 1 Q2：理解滑动窗口反而不如固定分块的场景——窗口重叠强制不相关句子合并，引入噪声
- Step 1 Q3：坦承重排序（Reranker）是之前没想到的概念，主动追问——这是学习者主动补全知识边界的行为
- Step 4 费曼：正确区分固定分块（FAQ、规则文档）和语义分块（论文、代码）的适用场景；准确诊断任务④中语义分块在 Tool Use 查询上失败的原因（"AI Agent 概念和 Tool Use 划分到一块，稀释了关联度"）

**亮点**：从任务④的实跑数据中提取出了语义分块的核心困境（阈值调参地狱），并用具体相似度数据支撑解释
**待加强**：重排序（Cross-Encoder）目前处于概念了解阶段，在后续模块 8（评估与监控）的 RAG 评估 pipeline 中会有动手实践的机会

### B - 代码实践能力：L3 🟢 (3.0)

**证据**：
- 任务①：独立完成检索对比循环，输出清晰展示了 chunk_size 与相似度的非单调关系
- 任务②：实现 `semantic_chunk()` 函数——核心算法正确（相邻句相似度 < threshold 时切分）。threshold=0.5 时产生 5 块，最接近"每块对应一个主题"。将函数提升到模块级别，展示了良好的代码组织意识
- 任务③：正确识别 Bug（`>=` → 应改为 `<`），诊断思路清晰——从相邻句相似度热力图出发分析异常输出
- 任务④：实现 `build_retriever()` 可配置检索器，成功在新文档上对比三种策略。开发生中遇到三个 Python 层面的小问题（chunk_size 默认值、kwargs 字典嵌套、切片 off-by-one），均在提示后快速修复

**亮点**：任务②中主动将 `semantic_chunk` 重构到模块级函数（原骨架中在 task2 内部），体现了"写出可复用代码"的意识
**待加强**：
- `**kwargs` 的传递机制（task ④ 中多包了一层 dict）——属于 Python 进阶语法，不计入 AI 概念考核
- 任务 ④ 中语义分块函数在新文档上的 threshold=0.4 可能不是最优值——建议后续尝试不同 threshold 跑对比

### C - 知识结构化能力：L3 🟢 (3.0)

**证据**：
- 笔记 `chunking.md` 覆盖：信号稀释原理、四种分块策略对比表、语义分块算法、三难权衡图、Bi-Encoder vs Cross-Encoder 对比、Chunk Overlap、Token 视角、实验发现与数据支撑
- 正确建立与 `[[embedding-basics]]` 和 `[[vector-database]]` 的关联——RAG 三环节（Embedding → 向量库 → 分块）形成闭环
- 实验发现部分将 Step 0 和 Step 3 的实跑数据整理为结构化结论，方便后续复习

## 薄弱点清单

1. **Cross-Encoder/Reranker 实操**（待学习）：目前处于概念层面，清楚"为什么需要"但未动手实现——预计在模块 8（RAG 评估 pipeline）中有机会实践
2. **语义分块阈值自动化**（轻度）：目前手动尝试 3 个阈值，生产环境中需要 grid search 或自适应阈值（如基于相邻句相似度分布的分位数）——可在模块 5 Agent 架构中学到自动化 pipeline 后回补
3. **Python **kwargs 传参机制**（工具层面，不计入 AI 考核）：task ④ 中误将 kwargs 嵌套了一层 dict 传递
