## 各维度评分

### 维度 A：概念理解深度 — L3 🟢（3.0 分）

**评估依据**：

| 来源 | 具体证据 |
|------|---------|
| Step 1 苏格拉底 | 问题 1：用「公司规章制度 vs 员工行为」类比 System Prompt vs User Prompt——抓住了优先级关系的本质。问题 2：正确推测是模型未响应而非约束冲突。问题 3：准确选择「身份层」为核心，理由是身份明确后行为规则和禁止行为会有隐式约束——展现了分层思维。 |
| Step 4 费曼复述 | 第一轮：将「好的 System Prompt」本质概括为「角色明确、行为规范明确、边界限制明确」——三层结构表述精准。第二问：从 token 生成角度解释了肯定式 > 否定式——「否定式告知不能怎么做，没说应该怎么做，对生成下一个 token 没有指导帮助」——抓住了生成机制的核心差异。 |
| 追问回答 | 被追问「世界顶级 Python 专家算不算好 System Prompt」时，正确识别「世界顶级」是模糊描述词而非行为激活信号——说明理解身份层的本质是「激活行为原型」而非「堆砌形容词」。 |

**达到该等级的理由**：

对三层架构、肯定式 vs 否定式的 token 生成机制、身份激活的本质有清晰理解。能从 LLM 内部机制（token 预测方向）解释提示工程的最佳实践，而非停留在经验规则层面。尚未达到 L4 是因为概念理解主要在学习过程中建立，尚未验证跨场景迁移能力（如不依赖提示独立分析一个生产级 System Prompt 的优劣）。

### 维度 B：代码实践能力 — L3 🟢（3.0 分）

**评估依据**：

| 任务 | 完成情况 | 亮点 / 问题 |
|------|---------|-----------|
| ① 角色修改 | ✅ 完成 | 三层结构完整（身份 + 先找问题再给建议 + 禁止笼统赞美）。语气观察精准——区分了「直接强硬」vs「先肯定再建议」。 |
| ② 面试官角色 | ✅ 完成 | **亮点**：量化约束（"至少2轮追问""最多1次提示"），将 Step 2 的「具体 > 模糊」原则直接应用。一次只问一个问题的设计避免了面试官"轰炸式追问"。 |
| ③ 修复 Bug | ✅ 完成 | 正确诊断三个问题：加身份、消矛盾（去"简短"留"详细"）、否定式改肯定式。小建议：最后三条仍用「禁止」开头，可进一步改为肯定式（如"只使用 requests 库和标准库"）。 |
| ④ 日报生成器 | ✅ 完成 | **亮点**：输出模板式设计（Markdown 结构 + 占位符 `[工作内容][耗时]`）让模型直接"填空"；边缘情况处理（"如无问题写无"）；工时表格排序规则。自发现 `max_tokens=500` 过小并修复——展现了工程调试意识。 |

**达到该等级的理由**：

4 个任务全部完成，质量递进明显：任务①在模板内填充 → 任务④从零设计完整 System Prompt。在脚手架支撑下展现了从「模仿」到「独立设计」的迁移能力。任务③的肯定式转换不完全（最后三条仍否定式）是细节层面问题，不影响对核心概念的掌握。

## 复习建议

### 短期
- 回顾任务 ③ 修复版本，尝试把所有「禁止」开头的指令改写成肯定式——验证肯定式是否比否定式更可靠
- 给任务 ④ 的日报生成器加一个边界测试：输入只含技术讨论、没有工作内容时，模型会编造日报还是会正确输出"无法生成"？

### 中期
- 模块 3 开始前回顾本主题的三层架构——Tool Definition 本质是用 JSON Schema 替代自然语言行为规则
- 思考：如果给一个 Agent 同时定义 System Prompt 和 Tool Definition，两者如何分工？System Prompt 管什么，Tool Definition 管什么？
