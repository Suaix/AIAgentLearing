# 学习记录 — 2026-07-03

## 今日学习内容

### 主题：结构化输出（JSON Mode）

- JSON Mode 的 API 层面约束机制：`response_format={'type': 'json_object'}` 在 token 采样时屏蔽不合法 token
- DeepSeek JSON Mode 特殊要求：Prompt 必须包含 "json" 关键字，否则 API 返回 400
- JSON Mode 的边界：保证合法 JSON 语法，但不保证 Schema 语义——需要客户端双重校验
- JSON Mode vs Tool Use：两个独立机制——前者约束输出格式，后者定义可调用工具
- 健壮的 JSON 解析：去 Markdown 包装 → 提取 JSON 子串 → 尾部逗号容错

- ③ 修复 JSON 解析 Bug：3 步容错（去 Markdown → 提取子串 → 尾部逗号）
  - 前后文字提取花费额外时间，属于字符串处理程序性知识的练习
- ④ 模板系统：`build_extraction_prompt()` 实现 Schema→Prompt 自动生成

## 遇到的问题与解决

1. **Step 1 问题 1 质疑**：指出实验 1A 的 system prompt 未写"以 JSON 格式输出"，模型是"恰好"输出 JSON——被确认正确，问题前提修正
2. **任务 ③ 前后文字提取**：`find()` + `rfind()` 组合逻辑需要仔细处理，涉及 min/max 和 -1 边界
3. **任务 ③ 逻辑运算符**：`or` vs `and` 在条件判断中的混淆（不影响功能）

## 关键收获

- JSON Mode 不是 Prompt 技巧，是 API 层硬约束——即使 temperature=2.0 也不会输出非法 JSON
- 生产环境标配：`response_format` + 客户端 Schema 校验 + 重试机制
- JSON Mode 和 Tool Use 是两个正交机制，交汇点在 Tool 的 arguments 格式
- 可复用的提取模板：Schema 定义 → 自动生成 Prompt → API 调用 → 校验——整条链路清晰
