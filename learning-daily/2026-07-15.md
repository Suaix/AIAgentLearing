# 学习记录 — 2026-07-15

## 今日学习内容

### 工具调用错误处理、超时与重试策略（Step 0~4）

- **错误分类**：可预判错误（参数格式、SQL 语法）vs 不可预判错误（API 宕机、网络超时）。LLM 能规避前者，后者才是错误处理真正的挑战。
- **错误流入 Agent 循环**：不是用 `try/except` 写死分支，而是把错误结果作为 tool role 消息喂回 LLM，让 LLM 读 error message 后自主决策。
- **双层保护机制**：软约束（System Prompt 规则引导优雅降级）+ 硬约束（`max_turns`、`call_with_timeout` 兜底防失控）必须共存。
- **外部超时包装 vs 内部降级链**：两者正交——timeout 管"会不会卡死"，降级链管"数据还能从哪拿"，可以叠加使用。
- **重试计数器按工具名隔离**：不同工具的可靠性独立，计数器必须用 `dict[str, int]` 而非单个 int。

### Python 语法的坑

- `ThreadPoolExecutor.submit(func(**kwargs))` → `TypeError: 'dict' is not callable`（参数和函数应分开传）
- `{val}` 被 Python 解析为 `set` 而非 `dict` → `not JSON serializable`（dict 必须有冒号）
- dict 的 `.get(key, default)` 方法防止 `KeyError`

## 遇到的问题与解决

1. **warmup 场景①③ LLM 直接绕过**：DeepSeek 能力太强，对可预判错误自动规避。解决：用随机概率替代 query 长度判断。
2. **task② `submit(func(**kwargs))` 报错**：参数和函数应分开传 → `submit(func, **kwargs)`。
3. **task② `except TimeoutError` 不精确**：应使用 `concurrent.futures.TimeoutError`。
4. **task③ KeyError**：用 `result["error"]`（错误类型）做 key 而非 `func_name`（工具名）。
5. **task③ `{err}` 被解析为 set**：`{k: v}` 才是 dict，`{v}` 是 set。
6. **task④ 降级概率太低看不到**：加 `_FORCE_FAIL_DEMO` 开关做强制演示。
7. **`.env` 加载不规范**：修正为 `load_dotenv()` 无参数方式。
