# 学习记录 — 2026-07-09

## 今日学习内容

- **System Prompt 设计**（模块 2 — Prompt Engineering，主题 5/6）
  - System Prompt 的 API 本质：`messages` 列表中 `role="system"` 的第一条消息，靠 RLHF 训练倾向和上下文位置优势起作用
  - 三层架构：身份定义 → 行为规则 → 禁止行为（三者缺一不可）
  - 肯定式指令 > 否定式指令：从 token 生成机制理解——肯定式给出方向，否定式只设障碍
  - System Prompt vs User Prompt 优先级及安全边界（提示注入防御）
  - 设计原则：肯定式 + 具体 + 量化可验证 + Markdown 结构化 + 覆盖边缘情况

- **Prompt 版本管理**（模块 2 — Prompt Engineering，主题 6/6，C 级参考）
  - 硬编码 Prompt 的三大痛点：版本混乱、全局更新困难、无法回滚
  - Prompt Registry 模式：集中式配置 + 变量模板 + 版本状态机（active/deprecated）
  - 变量模板的 DRY 原则：1 份模板 + N 份变量配置，避免 N 份几乎一样的 Prompt

## 遇到的问题与解决

1. **任务 ④ `max_tokens=500` 太小**：日报生成输出较长，500 token 截断了结果。修改为更大的值后正常。→ 工程体会：结构化长输出需要足够的 token 预算。
2. **任务 ③ 肯定式转换不彻底**：修复后仍保留 3 条「禁止」开头指令。→ 后续实践中有意识地做完全转换。
3. **任务 ② translator 缺少标点**：`{source_lang}翻译专家` 后缺少标点，与 `## 行为准则` 粘连。——提醒注意模板拼接时的格式完整性。
