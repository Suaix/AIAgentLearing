## 遇到的问题与解决

1. **f-string 冒号格式错误**：`f"{var:value}"` 中 `:` 被 Python 解释为格式说明符 → 改用 `f"角色: {var}"` 或直接打印
2. **工具传参错误**：最初用 LLM 回复文本作为工具参数 → 改为从分类阶段提取 `params` 字段
3. **路由 + 角色 System Prompt 冲突**：两个 system prompt 同时存在于 messages 中可能导致模型困惑 → 后续可考虑替换而非追加

## 费曼检验

- 六个主题各自解决什么：System Prompt（规范行为边界）→ Few-shot（提供示例参考）→ CoT（完善推理过程）→ ReAct（多轮观察行动）→ 结构化输出（规范输出格式）→ 版本管理（高效迭代对比）
- CoT 与 ReAct 的关系：CoT 是 ReAct 的思考组件，每一步 Action 前的 Thought 就是 CoT
- 六个主题相互独立，都是为了从不同维度规范 AI 行为和提升输出准确性

## 模块 3 开始：Tool Use / Function Calling — Tool Definition（Step 0~4）

### Step 0 热身

运行了 `tool_definition_warmup.py`，三个演示场景：
- ① 计算器工具：单工具调用 + 多轮顺序调用（先加后减）
- ② 天气查询：并行工具调用（一次调 2 个工具查北京+上海）
- ③ 多轮推理：查天气 → 模型推理出"需要带伞"建议

### Step 1-2 概念学习

核心概念：
- Tool Definition 三要素：name（标识）、description（LLM 判断调用的依据）、parameters（JSON Schema）
- Agent Loop：user → LLM 决策 → tool_calls? → Python 执行 → 结果返回 → 循环
- 并行 vs 顺序：LLM 根据依赖关系自主判断，无需开发者编程控制
- 消息历史管理：assistant(tool_calls) + tool(result) 两条消息都不能丢，tool_call_id 必须对应

### Step 3 引导式修改

四个任务全部完成：
- ① 改参数：temperature=0.0 vs 1.5，发现不影响工具选择只影响语气
- ② 加功能：完整实现 get_current_time 工具（Schema + ZoneInfo + 注册）
- ③ 修 Bug：修复除零守卫 + tool_call_id 缺失；识破误导性注释
- ④ 扩展场景：实现 query_product 工具 + 多工具组合查询

### Step 4 费曼检验

核心理解："给大模型一份工具使用说明(Schema)，大模型根据 description 语义匹配来决定调用哪些工具，执行后将结果返回，继续对话。不是关键词 if-else 匹配。"

### 关键发现

- LLM 能从工具返回的 error 中自我纠正（iPhone 16 → iPhone16 自动重试）
- `description` 质量直接决定调用成功率（macbook_pro 因不知命名规则而失败）
- 简单计算时 LLM 可能"心算"跳过 calculator 工具（效率策略）
