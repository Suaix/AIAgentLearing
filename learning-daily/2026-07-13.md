## 今日学习内容

### 模块 2 Step 5 — 独立里程碑：智能客服 Prompt 系统

从零构建了一个综合性的 Prompt 管理系统，涵盖：

1. **System Prompt 设计**：三个客服角色（通用客服、技术支持、投诉处理）× 两个版本（v1.0 简洁直接、v2.0 详细 SOP）
2. **Few-shot 示例库**：每角色 2-3 个精选示例
3. **CoT 推理**：tech_support 角色的 reasoning 必填，逐步排查
4. **ReAct 模式**：Thought → Action → Observation 循环，3 个模拟工具
5. **结构化 JSON 输出**：统一格式，容错处理
6. **Prompt 版本管理 + A/B 对比**：5 个测试用例 × 2 个版本，自动生成对比报告

### 架构设计

采用了「路由 Agent → 角色 Agent」的双层分发架构：
- 路由层：classify_query() 判断用户意图，分配到对应角色
- 执行层：run_react_agent() 加载角色 System Prompt，走 ReAct 循环

### 关键发现

- v1.0 和 v2.0 在本次测试中平分秋色（各 61/75 分）
- v2.0 的"谨慎"策略在测试 #5（产品对比）中反而导致模型不够自信
- Prompt 写得越"安全"，模型可能越保守——A/B 测试的核心价值在此

## 遇到的问题与解决

1. **f-string 冒号格式错误**：`f"{var:value}"` 中 `:` 被 Python 解释为格式说明符 → 改用 `f"角色: {var}"` 或直接打印
2. **工具传参错误**：最初用 LLM 回复文本作为工具参数 → 改为从分类阶段提取 `params` 字段
3. **路由 + 角色 System Prompt 冲突**：两个 system prompt 同时存在于 messages 中可能导致模型困惑 → 后续可考虑替换而非追加

## 费曼检验

- 六个主题各自解决什么：System Prompt（规范行为边界）→ Few-shot（提供示例参考）→ CoT（完善推理过程）→ ReAct（多轮观察行动）→ 结构化输出（规范输出格式）→ 版本管理（高效迭代对比）
- CoT 与 ReAct 的关系：CoT 是 ReAct 的思考组件，每一步 Action 前的 Thought 就是 CoT
- 六个主题相互独立，都是为了从不同维度规范 AI 行为和提升输出准确性
